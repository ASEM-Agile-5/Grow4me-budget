# TDD suite — GrowForMe Farm Budget

Unit tests and implementations for **critical** Farm Budget logic: financial calculations, input validation, and summary reporting. Built with **Jest** using **Test-Driven Development** (tests define behavior first; implementation follows).

---

## Prerequisites

- **Node.js** 18+ (LTS recommended)
- **npm** (comes with Node)

---

## Quick start

From the repository root:

```bash
cd docs/sprint-1/tdd
npm install
npm test
```

You should see **61 tests passed** and a coverage table. For continuous feedback while coding:

```bash
npm run test:watch
```

## What this folder contains


| Area                  | Source (`src/`)       | Tests (`tests/`)           | Role                                                                                                             |
| --------------------- | --------------------- | -------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| **Budget calculator** | `budgetCalculator.js` | `budgetCalculator.test.js` | Totals for expenses and revenue, profit, ROI, budget health labels (Healthy / Moderate / At Risk / Loss).        |
| **Input validator**   | `inputValidator.js`   | `inputValidator.test.js`   | Validates budgets, expenses, revenue rows, and user sign-up fields before data hits persistence or calculations. |
| **Budget summary**    | `budgetSummary.js`    | `budgetSummary.test.js`    | Builds the per-cycle summary object (totals, ROI, breakdown by expense category, target % achieved).             |


These are the pieces where bugs would **directly affect money figures** shown to farmers, so they are covered first with TDD.

---

## TDD workflow (Red → Green → Refactor)

1. **Red** — Add or change a test that describes the desired behavior; run tests and confirm failure.
2. **Green** — Implement the smallest change in `src/` that makes the test pass.
3. **Refactor** — Improve structure and names without changing behavior; tests must stay green.

For new critical logic, keep this order so the test file stays the **spec** for the module.

## Commands


| Command                                   | Purpose                                                        |
| ----------------------------------------- | -------------------------------------------------------------- |
| `npm test`                                | Run all tests with **coverage** (text + `coverage/lcov.info`). |
| `npm run test:watch`                      | Re-run tests on file save.                                     |
| `npx jest tests/budgetCalculator.test.js` | Run a single test file.                                        |
| `npx jest -t "calculateProfit"`           | Run tests whose name matches a pattern.                        |

## Coverage

- **Threshold:** Jest is configured in `package.json` to require **≥ 80%** global lines, functions, and branches. The build fails if coverage drops below that.
- **Reports:** After `npm test`, open `coverage/lcov-report/index.html` in a browser for a line-by-line report, or use `coverage/lcov.info` with your IDE or CI.

Typical numbers when everything passes: **~98% statements**, **~97% branches**, **100% functions/lines** on the three modules above.

## Adding a new critical module

1. Create `tests/<name>.test.js` with failing cases for the behavior you need.
2. Add `src/<name>.js` and export only what the tests require.
3. Run `npm test` until green and coverage stays above the threshold.
4. Wire the module into your app’s API or UI layer in the main repo when ready.

## Troubleshooting


| Issue                             | What to try                                                                                                                            |
| --------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| `Cannot find module '../src/...'` | Run tests from the `tdd` directory, or fix relative paths in the test file.                                                            |
| Coverage threshold failure        | Add tests for new branches or uncovered lines, or adjust only if the team agrees to change `jest.coverageThreshold` in `package.json`. |
| `jest: command not found`         | Run `npm install` inside `tdd`.                                                                                                        |
